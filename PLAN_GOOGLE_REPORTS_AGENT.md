# План: Агент Побудови Google Sheets Для Cashflow І P&L

## Мета

Побудувати окремий модуль, який після завершення поточного опитування:

1. бере збережені дані користувача з БД;
2. створює папку на Google Drive з назвою на основі Telegram-акаунта;
3. створює в цій папці потрібні таблиці Google Sheets;
4. заповнює їх за ТЗ;
5. перевіряє, що структура, формули і статті створені коректно;
6. відкриває доступ по посиланню;
7. повертає користувачу готові посилання.

Робочий підхід: файли створюються в сервісному Google Drive або у виділеній папці сервісного акаунта, а користувач отримує доступ по посиланню і за потреби завантажує або копіює собі.

## Бажаний Результат

Для кожного користувача створюється окрема папка виду:

`@username - Financial Reports`

або, якщо username відсутній:

`tg_<telegram_id> - Financial Reports`

Всередині папки створюються:

1. `Cashflow`
2. `P&L`
3. за потреби пізніше: `Balance`, `Dashboard`, `Unit Economics`, `Instructions`

## Обрана Модель Доступу

Перший етап реалізації: `Service Account`.

Причина:

1. не треба авторизовувати кожного користувача через Google;
2. простіше автоматизувати на сервері;
3. можна централізовано створювати папки і таблиці;
4. можна одразу видавати доступ по посиланню.

Що це означає:

1. Google Drive/Sheets створюються від сервісного акаунта;
2. файли лежать у просторі сервісного акаунта або в спільній папці;
3. доступ користувачу дається як `Anyone with the link` або через email пізніше.

## Архітектура

Рекомендується винести логіку в окремі модулі:

1. `src/google/auth.js`
2. `src/google/drive.js`
3. `src/google/sheets.js`
4. `src/google/reportBuilder.js`
5. `src/google/reportValidator.js`
6. `src/google/reportTemplates.js`

### Відповідальність Модулів

`auth.js`

1. ініціалізація Google credentials;
2. створення клієнтів для Drive і Sheets API.

`drive.js`

1. створення папки користувача;
2. пошук існуючої папки;
3. встановлення доступу по посиланню;
4. побудова посилань на папку і документи.

`sheets.js`

1. створення spreadsheet;
2. додавання аркушів;
3. запис заголовків, формул і даних;
4. форматування;
5. freeze rows/columns;
6. data validation;
7. batchUpdate операції.

`reportTemplates.js`

1. шаблон Cashflow;
2. шаблон P&L;
3. конфіг заголовків, секцій, кольорів, формул.

`reportBuilder.js`

1. оркестрація створення папки і таблиць;
2. побудова таблиць з JSON-моделі;
3. повернення фінального набору посилань.

`reportValidator.js`

1. перечитування створених таблиць назад через API;
2. перевірка структури;
3. перевірка формул;
4. перевірка повноти статей;
5. повернення звіту про успіх або помилки.

## Логіка Створення Папки

### Назва Папки

Правило:

1. якщо є `telegram username` -> `@username - Financial Reports`
2. якщо username немає -> `tg_<telegram_id> - Financial Reports`

Опціонально:

1. додавати дату створення;
2. додавати назву бізнесу;
3. нормалізувати заборонені символи.

Приклад:

`@matsuk_client - Financial Reports`

або

`tg_123456789 - Financial Reports`

### Поведінка При Повторному Запуску

Щоб не плодити дублікати, потрібна стратегія:

1. спочатку перевіряємо в БД, чи вже є `google_drive_folder_id`;
2. якщо є, використовуємо існуючу папку;
3. якщо немає, створюємо нову;
4. якщо папка втрачена або видалена, створюємо заново і оновлюємо БД.

## Що Потрібно Дозберегти В БД

Бажано додати окремі поля або JSON-блок типу `google_reports`:

```json
{
  "folder_id": "...",
  "folder_url": "...",
  "cashflow_sheet_id": "...",
  "cashflow_url": "...",
  "pl_sheet_id": "...",
  "pl_url": "...",
  "last_build_status": "success",
  "last_build_error": "",
  "last_validated_at": "2026-04-10T18:00:00Z"
}
```

Це дає змогу:

1. не будувати все з нуля кожного разу;
2. перевіряти стан попередньої побудови;
3. перегенеровувати окремо Cashflow або P&L;
4. віддавати користувачу повторно ті самі посилання.

## Валідація Після Побудови

Після створення файлів агент має сам себе перевірити.

### Перевірки Для Cashflow

1. створений spreadsheet;
2. є потрібні аркуші;
3. усі статті перенесені;
4. немає порожніх ключових колонок;
5. частота і регулярність відображені коректно.

### Перевірки Для P&L

1. всі секції створені;
2. всі статті потрапили у правильну секцію;
3. формули рівнів прибутку записані;
4. немає загублених статей;
5. заголовки і структура відповідають шаблону.

### Формат Результату Валідатора

```json
{
  "valid": true,
  "folder_created": true,
  "cashflow_created": true,
  "pl_created": true,
  "checks": [
    "folder_exists",
    "cashflow_sheets_exist",
    "pl_sections_exist",
    "all_items_mapped"
  ],
  "errors": []
}
```

## Режим Роботи Агента

Краще не давати ШІ прямий сирий доступ до Google API без обмежень.

Безпечна модель:

1. звичайний код на Node.js виконує всі API-виклики;
2. ШІ тільки формує структуровані інструкції для builder-а;
3. builder викликає дозволений набір функцій;
4. validator перевіряє результат;
5. якщо перевірка не пройдена, запускається controlled retry.

Тобто ШІ тут є оркестратором, а не прямим джерелом довільних API-запитів.

## Потрібні Google Налаштування

### Google Cloud

1. створити project;
2. увімкнути `Google Drive API`;
3. увімкнути `Google Sheets API`;
4. створити `Service Account`;
5. згенерувати JSON key;
6. покласти ключ у змінну середовища або secure file storage.

### Змінні Середовища

Приклад:

```env
GOOGLE_SERVICE_ACCOUNT_EMAIL=...
GOOGLE_SERVICE_ACCOUNT_PRIVATE_KEY=...
GOOGLE_DRIVE_PARENT_FOLDER_ID=...
GOOGLE_REPORTS_SHARE_MODE=anyone_with_link
```

Краще зберігати ключ або як base64, або як JSON в secret manager, а не у відкритому файлі репозиторію.

## Етапи Реалізації

### Етап 1. Підготовка Даних

1. зафіксувати остаточний формат `financial_reports_model`;
2. вирішити, які поля підуть у Cashflow, а які в P&L;
3. додати `google_reports` в БД.

### Етап 2. Підключення Google API

1. підняти credentials;
2. написати `auth.js`;
3. написати базові обгортки над Drive/Sheets.

### Етап 3. Створення Папки Користувача

1. побудова імені папки з username або telegram_id;
2. створення/пошук папки;
3. збереження `folder_id` і `folder_url` у БД.

### Етап 4. Побудова Cashflow

1. створення таблиці;
2. створення аркушів;
3. запис статей;
4. додавання форматування і формул;
5. валідація.

### Етап 5. Побудова P&L

1. створення таблиці;
2. створення секцій;
3. запис статей по `pl_structure`;
4. формули рівнів прибутку;
5. валідація.

### Етап 6. Інтеграція З Поточним Ботом

1. після завершення блоку 2 додати кнопку `Створити мої таблиці`;
2. викликати builder;
3. після успіху віддати:
   - посилання на папку;
   - посилання на Cashflow;
   - посилання на P&L.

### Етап 7. Повторне Використання

1. якщо таблиці вже існують, бот пропонує:
   - оновити існуючі;
   - створити нові;
   - просто відкрити папку.

## Критерії Готовності

Рішення вважається готовим, коли:

1. для будь-якого користувача створюється окрема папка;
2. в папці будуються `Cashflow` і `P&L`;
3. доступ працює по посиланню;
4. всі статті з БД потрапляють у таблиці без втрат;
5. validator підтверджує цілісність структури;
6. бот повертає фінальні посилання без ручного втручання.

## Рекомендація По Реалізації

Найкращий порядок робіт:

1. спочатку зробити builder без ШІ, на жорсткому шаблоні;
2. коли шаблон стабільно будується, додати ШІ тільки для оркестрації і перевірки;
3. не починати з повністю автономного агента, який "сам усе вирішує".

Це зменшить ризик помилок і сильно прискорить перший робочий запуск.
