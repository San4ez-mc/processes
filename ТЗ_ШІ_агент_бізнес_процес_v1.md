## 1. Загальний опис системи

Агент — це окремий модуль Telegram-бота, який активується на уроці 1.3 курсу. Клієнт проходить інтерв'ю в режимі діалогу: бот задає питання по черзі, клієнт відповідає текстом або вибирає варіант. На виході — перевірена swimlane-схема у форматі Mermaid, яка зберігається в базі бота і використовується в усіх наступних уроках.

### Мета агента

- Зібрати повний бізнес-процес компанії клієнта
- Структурувати його у вигляді swimlane по ролях
- Перевірити логіку і повноту схеми
- Повернути клієнту схему в зрозумілому вигляді
- Зберегти JSON-модель для наступних уроків курсу

### Хто використовує

- Власник малого бізнесу (7+ людей у команді)
- Будь-який рівень технічної грамотності
- Не знає що таке swimlane або Mermaid
- Проходить курс самостійно у своєму темпі
- Хоче отримати схему не малюючи руками

## 2. Архітектура системи

Система складається з трьох послідовних компонентів, які передають дані один одному через спільний JSON-стан сесії.

```
[Клієнт у Telegram]
        ↓
[Компонент 1: Інтерв'ю-агент]  →  наповнює JSON process_model
        ↓ ###INTERVIEW_COMPLETE###
[Компонент 2: Валідатор]  →  перевіряє логіку JSON
        ↓ valid: true
[Компонент 3: Mermaid-генератор]  →  генерує код схеми
        ↓
[Рендер mermaid-cli → PNG]
        ↓
[Клієнт отримує фото схеми]
```

### 2.1 Компонент 1 — Інтерв'ю-агент

Веде діалог з клієнтом. Задає питання, обробляє відповіді, наповнює JSON-модель процесу. Реалізується як системний промпт з пам'яттю сесії.

**Технічні вимоги:**

- Модель: `claude-sonnet-4-20250514` або `gpt-4o` через API
- Пам'ять: зберігати повну історію діалогу в сесії
- Стан: JSON-об'єкт `process_model` оновлюється після кожної відповіді
- Валідація вхідних даних: перевіряти що відповідь не порожня і релевантна
- Fallback: якщо клієнт відповів нечітко — уточнювальне питання, максимум 2 рази

### 2.2 Компонент 2 — Валідатор логіки

Окремий промпт-агент, який отримує заповнений JSON і перевіряє його на помилки. Запускається після завершення інтерв'ю, до генерації Mermaid.

**Що перевіряє валідатор:**

- Наявність стартової точки (тригер залучення клієнта)
- Наявність кінцевої точки (виконання зобов'язань / завершення угоди)
- Кожен блок має мінімум одне вхідне і одне вихідне з'єднання
- Жодна swimlane не є порожньою — є хоча б одна дія
- Немає «висячих» блоків без з'єднань
- Назви блоків — дієслово + об'єкт (наприклад «Надіслати КП», не просто «КП»)
- Логічна послідовність: не може бути дії після завершення
- Немає циклічних залежностей без умовного розгалуження

> Якщо знайдені помилки — агент **не** генерує схему, а повертається в діалог з конкретним уточнюючим питанням. Максимум 3 ітерації виправлень.

### 2.3 Компонент 3 — Mermaid-генератор

Отримує валідний JSON і генерує Mermaid-код у форматі flowchart з swimlanes. Бот рендерить код у PNG через бібліотеку `mermaid-cli` на сервері і відправляє клієнту.

**Вимоги до Mermaid-коду:**

- Формат: `flowchart TD` або `LR` (залежно від кількості swimlanes)
- Кожна swimlane — окремий `subgraph` з назвою ролі
- Блоки: прямокутники для дій, ромби для рішень, кола для старту/фінішу
- З'єднання: стрілки з підписами якщо є умова (Так/Ні)
- Назви блоків: коротко, максимум 4 слова
- Рендер: mermaid-cli → PNG 1200px ширина, білий фон

## 3. JSON-модель бізнес-процесу

Центральна структура даних. Заповнюється поступово під час інтерв'ю. Передається між компонентами. Зберігається в БД бота після завершення.

```json
{
  "session_id": "uuid",
  "business_type": "послуги | продукт | виробництво | торгівля",
  "team_size": 12,
  "lanes": [
    {
      "id": "lane_marketing",
      "role": "Маркетинг",
      "responsible": "Маркетолог Іванова",
      "nodes": [
        {
          "id": "n1",
          "type": "start",
          "label": "Запуск реклами",
          "description": "Facebook + Google Ads"
        },
        {
          "id": "n2",
          "type": "action",
          "label": "Отримати заявку",
          "description": "Заявка через форму на сайті"
        }
      ]
    },
    {
      "id": "lane_sales",
      "role": "Продажі",
      "responsible": "Менеджер Петренко",
      "nodes": [
        {
          "id": "n3",
          "type": "action",
          "label": "Зателефонувати клієнту",
          "description": ""
        },
        {
          "id": "n4",
          "type": "decision",
          "label": "Клієнт зацікавлений?",
          "description": ""
        }
      ]
    }
  ],
  "edges": [
    { "from": "n1", "to": "n2", "label": "" },
    { "from": "n2", "to": "n3", "label": "" },
    { "from": "n4", "to": "n5", "label": "Так" },
    { "from": "n4", "to": "n_end_no", "label": "Ні" }
  ],
  "status": "draft | validated | complete",
  "validation_errors": [],
  "mermaid_code": ""
}
```

### Типи вузлів (`node.type`)

| Тип | Синтаксис Mermaid | Опис |
|---|---|---|
| `start` | `([Назва])` | Початок процесу. Один на весь процес |
| `end` | `([Назва])` | Завершення. Може бути кілька (успіх / відмова) |
| `action` | `[Назва]` | Дія. Основний тип |
| `decision` | `{Назва?}` | Розгалуження. Завжди 2+ вихідних ребра з підписами |
| `subprocess` | `[[Назва]]` | Підпроцес для складних дій |

## 4. Сценарій діалогу

Агент веде клієнта через 6 тематичних блоків. Кожен блок — серія питань по одній темі. Питання задаються **по одному**. Агент не переходить до наступного поки поточне не закрите.

### Блок 0 — Контекст бізнесу (2–3 питання)

**Мета:** зрозуміти тип бізнесу перед тим як ставити специфічні питання.

1. «Розкажіть коротко — чим займається ваша компанія? Що ви продаєте або яку послугу надаєте?»
2. «Скільки людей у вашій команді зараз? (включаючи підрядників і неповний робочий день)»
3. «Як би ви описали свій бізнес: це послуги, фізичний продукт, виробництво чи щось інше?»

→ *Результат: заповнюються поля `business_type` і `team_size` в JSON*

---

### Блок 1 — Залучення клієнта (3–4 питання)

**Мета:** визначити як бізнес знаходить клієнтів і хто за це відповідає.

1. «З чого починається ваш бізнес-процес? Як потенційний клієнт взагалі дізнається про вас?»
2. «Хто в команді відповідає за залучення клієнтів — є окрема людина чи ви самі?»
3. «Як клієнт залишає заявку або виходить на контакт? Через сайт, телефон, соцмережі?»
4. «Що відбувається одразу після того як прийшла заявка — хто її бачить першим?»

→ *Результат: створюється перша swimlane (Маркетинг або Власник), додається стартовий вузол*

---

### Блок 2 — Продаж і кваліфікація (3–5 питань)

**Мета:** описати шлях від заявки до підписання угоди / першої оплати.

1. «Що відбувається після отримання заявки? Хто і як зв'язується з клієнтом?»
2. «Як ви розумієте що клієнт підходить вам — є якась перевірка або кваліфікація?»
3. «Як виглядає ваша комерційна пропозиція — ви надсилаєте КП, прайс, робите кошторис?»
4. «Що відбувається якщо клієнт відмовляє або просить час подумати?»
5. «Коли і як клієнт дає згоду — підписує договір, надсилає підтвердження, платить?»

→ *Результат: swimlane Продажі, вузли кваліфікації та рішення про угоду*

---

### Блок 3 — Виконання і виробництво (3–5 питань)

**Мета:** описати основну операційну частину — як створюється цінність для клієнта.

1. «Після того як клієнт погодився — що відбувається далі? Хто починає роботу?»
2. «Опишіть основні кроки виконання замовлення або надання послуги — по порядку»
3. «Хто ще в команді залучений на цьому етапі крім менеджера продажів?»
4. «Є якісь точки контролю якості або погодження з клієнтом по ходу роботи?»
5. «Що відбувається якщо щось пішло не так або клієнт незадоволений?»

→ *Результат: swimlanes виробничих ролей, основні операційні вузли*

---

### Блок 4 — Оплата і закриття (2–3 питання)

**Мета:** зафіксувати як відбувається оплата і закриття угоди.

1. «Як і коли клієнт платить — передоплата, постоплата, частинами? Хто виставляє рахунок?»
2. «Хто в команді контролює що оплата надійшла?»
3. «Що є фінальним кроком для клієнта — підписаний акт, відгук, закрита заявка?»

→ *Результат: вузли оплати та фінальні вузли `end` в JSON*

---

### Блок 5 — Команда і ролі (2–3 питання)

**Мета:** перевірити що всі ролі вказані і swimlanes повні.

1. «Ми описали такі ролі: [перелік з JSON]. Хтось ще бере участь у процесі кого ми не згадали?»
2. «Є у вас бухгалтер або людина яка веде фінансовий облік? Коли вона залучається?»
3. «Хто відповідає за фінансову звітність у вас зараз?»

→ *Результат: фінальна перевірка повноти ролей, додавання swimlane Фінанси якщо відсутня*

## 5. Системні промпти

### SYSTEM PROMPT — Інтерв'ю-агент

```
Ти — ШІ-асистент курсу "Фінансова система малого бізнесу".
Твоя задача — провести інтерв'ю з власником бізнесу і побудувати
JSON-модель бізнес-процесу його компанії.

## ПРАВИЛА ДІАЛОГУ
- Задавай ОДНЕ питання за раз. Ніколи не задавай два питання в одному повідомленні.
- Після відповіді клієнта — коротко підтвердь що зрозумів (1 речення) і задай наступне.
- Говори простою мовою. Ніяких термінів типу "swimlane", "вузол", "JSON".
- Якщо відповідь нечітка — уточни. Максимум 2 уточнення на одне питання.
- Якщо клієнт відповів не по темі — м'яко поверни до питання.
- Тон: дружній, підтримуючий, як досвідчений консультант.

## СТРУКТУРА ПРОЦЕСУ
Ти будуєш процес від залучення клієнта до виконання зобов'язань.
Процес завжди починається з того як клієнт дізнається про бізнес.
Процес завжди завершується виконанням замовлення і закриттям оплати.

## JSON-СТАН
Після кожної відповіді ти оновлюєш внутрішній JSON process_model.
Цей JSON НЕ показуєш клієнту — він тільки для системи.
Коли всі блоки заповнені — виводиш маркер: ###INTERVIEW_COMPLETE###

## ПОТОЧНИЙ СТАН СЕСІЇ
{{process_model_json}}

## ПРОГРЕС
Поточний блок: {{current_block}}
Завершені блоки: {{completed_blocks}}
```

### SYSTEM PROMPT — Валідатор

```
Ти — валідатор бізнес-процесу. Отримуєш JSON-модель процесу.
Твоя задача — знайти логічні помилки і незавершені місця.

## ЩО ПЕРЕВІРЯТИ
1. Є рівно один вузол типу "start"
2. Є хоча б один вузол типу "end"
3. Кожен вузол (крім start) має мінімум одне вхідне ребро
4. Кожен вузол (крім end) має мінімум одне вихідне ребро
5. Вузол типу "decision" має рівно 2+ вихідних ребра з підписами
6. Жодна swimlane не порожня
7. Є swimlane з фінансовою функцією (бухгалтер/фінансист)
8. Є вузол пов'язаний з отриманням оплати

## ФОРМАТ ВІДПОВІДІ
Якщо помилок немає:
{ "valid": true, "errors": [] }

Якщо є помилки:
{
  "valid": false,
  "errors": [
    {
      "code": "MISSING_PAYMENT_NODE",
      "description": "Не знайдено вузол отримання оплати",
      "question_to_ask": "Як і коли клієнт платить вам — передоплата чи після виконання?"
    }
  ]
}
```

### SYSTEM PROMPT — Mermaid-генератор

```
Ти — генератор Mermaid-коду. Отримуєш валідний JSON бізнес-процесу.
Генеруй ТІЛЬКИ Mermaid-код, без пояснень, без markdown-обгортки.

## ПРАВИЛА ГЕНЕРАЦІЇ
- Формат: flowchart TD
- Кожна lane → окремий subgraph з id="lane_{id}" і назвою ролі
- start  → ([Назва])
- end    → ([Назва])
- action → [Назва]
- decision → {Назва?}
- subprocess → [[Назва]]
- ID вузлів: використовуй node.id з JSON без змін
- Ребра: --> для звичайних, -->|Так| і -->|Ні| для decision
- Назви: максимум 4 слова, скорочуй якщо довше

## ПРИКЛАД ВИВОДУ

    n5[Надіслати КП]
    n_end_no([Відмова])
  end
  n1 --> n2 --> n3 --> n4
  n4 -->|Так| n5
  n4 -->|Ні| n_end_no
```

## 6. Логіка Telegram-бота (псевдокод)

```javascript
async function handleMessage(user_id, text) {

  const session = await db.getSession(user_id)

  if (session.status === 'complete') {
    return sendMessage('Ваш процес вже побудований. /restart щоб почати заново')
  }

  // Додаємо повідомлення в історію
  session.history.push({ role: 'user', content: text })

  // Запит до LLM (інтерв'ю-агент)
  const response = await callLLM({
    system: INTERVIEW_PROMPT
      .replace('{{process_model_json}}', JSON.stringify(session.process_model))
      .replace('{{current_block}}', session.current_block)
      .replace('{{completed_blocks}}', session.completed_blocks.join(', ')),
    messages: session.history
  })

  // Оновлюємо JSON-модель з відповіді агента
  const updated_model = extractJSON(response)
  if (updated_model) {
    session.process_model = updated_model
  }

  // Перевіряємо чи завершено інтерв'ю
  if (response.includes('###INTERVIEW_COMPLETE###')) {

    // Запускаємо валідатор
    const validation = await callLLM({
      system: VALIDATOR_PROMPT,
      messages: [{ role: 'user', content: JSON.stringify(session.process_model) }]
    })

    if (!validation.valid) {
      // Повертаємось в діалог з уточнюючим питанням
      session.validation_attempts += 1
      if (session.validation_attempts > 3) {
        await sendMessage(user_id, 'Схема побудована з попередженнями. Ось ваш процес:')
        // далі генеруємо попри помилки
      } else {
        return sendMessage(user_id, validation.errors[0].question_to_ask)
      }
    }

    // Генеруємо Mermaid
    const mermaid_code = await callLLM({
      system: MERMAID_PROMPT,
      messages: [{ role: 'user', content: JSON.stringify(session.process_model) }]
    })

    // Рендеримо PNG
    const png_buffer = await renderMermaid(mermaid_code)

    // Зберігаємо і відправляємо
    session.mermaid_code = mermaid_code
    session.status = 'complete'
    await db.saveSession(session)

    await sendPhoto(user_id, png_buffer)
    await sendMessage(user_id, COMPLETION_MESSAGE)
    return
  }

  // Звичайна відповідь агента
  const bot_text = extractText(response)
  session.history.push({ role: 'assistant', content: bot_text })
  await db.saveSession(session)
  await sendMessage(user_id, bot_text)
}
```

## 7. Рендер Mermaid → PNG

### Встановлення

```bash
npm install @mermaid-js/mermaid-cli
```

### Функція рендеру (Node.js)

```javascript
const { run } = require('@mermaid-js/mermaid-cli')
const fs = require('fs')
const path = require('path')

async function renderMermaid(mermaid_code) {
  const tmpDir = '/tmp/mermaid'
  const inputFile = path.join(tmpDir, 'diagram.mmd')
  const outputFile = path.join(tmpDir, 'diagram.png')

  fs.mkdirSync(tmpDir, { recursive: true })
  fs.writeFileSync(inputFile, mermaid_code)

  await run(inputFile, outputFile, {
    puppeteerConfig: { args: ['--no-sandbox'] },
    mermaidConfig: {
      theme: 'default',
      flowchart: { curve: 'basis' }
    },
    width: 1200,
    backgroundColor: 'white'
  })

  const buffer = fs.readFileSync(outputFile)
  fs.unlinkSync(inputFile)
  fs.unlinkSync(outputFile)
  return buffer
}
```

## 9. Зберігання даних

### Схема таблиці `sessions`

```sql
CREATE TABLE sessions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  telegram_id     BIGINT UNIQUE NOT NULL,
  process_model   JSONB,
  mermaid_code    TEXT,
  history         JSONB,
  status          VARCHAR(20) DEFAULT 'draft',  -- draft | validated | complete
  validation_attempts INT DEFAULT 0,
  created_at      TIMESTAMP DEFAULT NOW(),
  updated_at      TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_sessions_telegram_id ON sessions(telegram_id);
```

### Вимоги до БД

- PostgreSQL (рекомендовано) або MongoDB
- `process_model` і `history` — JSONB для швидкого пошуку
- `process_model` доступний іншим модулям курсу через `telegram_id`
- Резервне копіювання раз на добу
- Шифрування персональних даних (GDPR)
